Introduction

Transformers are amazing at remembering everything, but they’re also that friend who insists on reading every single group chat message before replying. State-space models (SSMs) are the opposite: fast, linear-time, and chill, but they stuff everything into a fixed-size box and hope the right bits fall out when you shake it.

Latent-bottleneck models like Perceiver and byte-level chunkers sit in the middle. They say: let’s first compress the input into a smaller latent workspace, and then do the heavy attention inside that tiny room. Neat! But almost always, the size of that room is picked once and never touched again.

This blog post is about treating that latent workspace like a budget you can spend over time, not a fixed law of the universe.

Concretely, we:

  • Build a Super Transformer: a decoder-only model whose input sequence is explicitly split into prefix, latent, and decoder regions.
  • Use Super Attention to wire these regions together with a flexible mask.
  • Train on the copy task with a linear latent schedule:
    • start with many latent tokens,
    • linearly decay the latent count from a large value $L_{\max}$ down to a small value $L_{\min}$ over training.

The punchline:

At the same final latent budget $L_{\min}$, linear decay of latent count during training gives lower copy-task loss than training with $L=L_{\min}$ from scratch.

That’s it. No 30-page ablation zoo. Just one simple experiment, a clear win, and some connections to the rest of the efficient sequence modeling menagerie.

Background: Latent Bottlenecks, State Size, and Compute

A lot of modern sequence models sneak in a bottleneck somewhere:

  • Perceiver / Perceiver IO / Perceiver AR: map high-dimensional or long inputs into a fixed-size latent array via cross-attention, then do all the heavy lifting on that latent.
  • BLT, H-Net, H-Net++: learn to chunk bytes into higher-level units (patches, chunks-of-chunks) and attend over those instead of every byte.
  • S4, Mamba: keep a fixed-dimensional state vector per layer and update it with learned dynamics, effectively summarizing the whole past in that state.

Each of these picks a state size:

  • SSMs: state dimension (e.g., 256, 512, 1024).
  • Latent models: number of latent tokens.
  • Standard Transformers: sequence length itself.

Typically, that choice is made once (“we use 128 latents”) and never revisited.

In contrast, the Super Transformer is built so that latent size is a first-class knob: you can change it for each sample!

The Super Transformer Architecture

The Super Transformer is a decoder-only prefix LM with three regions in a single 1D sequence:

  1. Prefix tokens (mask = 1): the “things we want to remember.”
  2. Latent tokens (mask = 2): the variable-size workspace.
  3. Decoder tokens (mask = 3): where we generate outputs (teacher-forced during training).

You can picture a packed example as:

[prefix .........]  [LAT LAT LAT ...]  [<GO> y1 y2 ...]
      1                     2                  3

All tokens go through the same stack of Transformer blocks. What matters is who can see whom via the attention mask.

Token layout and masks

We use a mask callback with flex_attention that enforces the following rules inside each segment:

  • Prefix and latent tokens form a global, bidirectional workspace.
  • Latent tokens have full self-attention within the latent region.
  • Decoder tokens:
    • are causal within the decoder region,
    • can read from latents,
    • cannot write back into prefix or latents.

In pseudocode (simplified for the blog):

def mask_mod(b, h, q_idx, kv_idx):
    causal  = (q_idx >= kv_idx)
    sameseg = (segment_ids[b, q_idx] == segment_ids[b, kv_idx])

    prefix_q = (attention_mask[b, q_idx] == 1)
    prefix_k = (attention_mask[b, kv_idx] == 1)
    latent_q = (attention_mask[b, q_idx] == 2)
    latent_k = (attention_mask[b, kv_idx] == 2)
    dec_q    = (attention_mask[b, q_idx] == 3)
    dec_k    = (attention_mask[b, kv_idx] == 3)

    # Prefix and latents: fully mixed workspace
    prefix_cross = (prefix_q & latent_k) | (latent_q & prefix_k)
    latent_self  = latent_q & latent_k

    # Decoder: causal + can read latents
    dec_self  = dec_q & dec_k & causal
    dec_cross = dec_q & latent_k

    section = prefix_cross | latent_self | dec_self | dec_cross
    return section & sameseg

All of the fun stuff (prefix encoding, memory, retrieval) happens inside this structured layout. Importantly, nowhere in this code do we bake in “there must be exactly 32 latents.” We just give it whatever latent block the data prep decided to insert.

Super Attention as a compute dial

Standard self-attention on a sequence of length $T$ costs $O(T^2)$. Super Attention thinks about it differently:

  • Let $T = P + L + D$ be prefix + latent + decoder length.
  • We care most about the cost of attention inside the latent block, which is $O(L^2)$.
  • By changing $L$, we change how expensive (and how expressive) that core workspace is.

This lets us run in different regimes:

  • Sub-quadratic-ish: small $L$, larger $P$; most cost is still dominated by the short latent sequence.
  • Quadratic: moderate $L$; the latent workspace is the main bottleneck.
  • “Super” regimes (conceptually): if you let $L$ grow and stack many dense latent layers, you can spend more than a standard Transformer would—on purpose, on a much smaller state.

This post focuses on one simple use of that dial: linearly decaying $L$ during training on the copy task.

Copy Task Experiments with a Linear Latent Schedule

The copy task is the classic sequence-modeling cringe test: can your model see a random sequence and reproduce it later, token for token?

Here’s how we use it:

  • The prefix region holds the sequence to memorize.
  • The latent region is the memory buffer.
  • The decoder region is trained (with teacher forcing) to output the same sequence.

If the model learns to:

  • write the prefix into latents, and
  • read it back into the decoder,

then copy loss drops.

If the latent workspace is too small or too poorly trained, the copy loss stays high.

Training regimes

Let $L$ be the number of latent tokens per example.

We compare three regimes, all using the same model architecture and hyperparameters and all ending at the same small latent budget $L_{\min}$:

  1. Fixed-small (train-small-from-scratch)
    • Use $L = L_{\min}$ at every training step.
  2. Fixed-large (train-large)
    • Use $L = L_{\max}$ at every training step.
  3. Linear latent schedule (ours)
    • At step $t$ (out of $T$ total steps), set [ L(t) = L_{\max} - \left\lfloor (L_{\max} - L_{\min}) \cdot \frac{t}{T} \right\rfloor. ]
    • In words: start at $L_{\max}$, then linearly decay latent count down to $L_{\min}$ across training.

All of these end in the same small-latent model; the difference is the path they took to get there.

Results

The picture is simple and nice:

The linear latent schedule converges to a lower copy loss than training with $L=L_{\min}$ from scratch.

The fixed-large model is there mainly as a sanity check (it has the easiest life). The interesting comparison is between fixed-small and linear schedule: same destination, different journeys, better outcome for the scheduled one.

Intuition for why it helps

A rough story that seems consistent with what we see:

  • Early on, give the model a big desk.
    With $L_{\max}$ latents, the model can stash information all over the place and still learn something useful about how to encode and retrieve the prefix.

  • Later, ask it to tidy up.
    As $L$ decays linearly to $L_{\min}$, the model is gently forced to compress those behaviors into fewer slots. But it’s compressing an already-functional strategy, not groping blindly in a low-capacity space from step 0.

  • Attention is good at consolidating.
    Attention-based routing from prefix to latents can start off redundant and then gradually learn to merge multiple latent “roles” into fewer tokens as capacity shrinks. A fixed-small model has to make those choices while everything is still noisy.

This is basically progressive compression but for the latent workspace instead of the weight matrix.

Positioning Among Other Sequence Models

Where does this fit in the zoo of sequence models?

  • RNNs and LSTMs.
    They maintain a hidden state updated step-by-step. Great for streaming, but training on long sequences is tricky and parallelism is limited. The Super Transformer trades recurrence for parallel attention, and trades a vector state for a small set of latent tokens.

  • Strongly-typed RNNs.
    These enforce structure on hidden state updates to make optimization more stable. The Super Transformer takes a different angle: it enforces structure at the sequence layout level (prefix/latent/decoder) and leaves the internal block as attention + MLP.

  • S4 and Mamba.
    S4 and Mamba are SSM-based and shine on very long, often continuous sequences. They compress history into a fixed-dimensional state with a carefully designed update rule. The Super Transformer is more like “a handful of attention-based memory slots”: a small state, but with content-addressable read/write. On copy-like tasks, that buys it a lot.

  • BLT, H-Net, H-Net++.
    These learn the segmentation of raw bytes into higher-level units (patches, chunks). Once you have a compressed sequence, you still need to process it. Our story starts there: we assume a latent bottleneck exists and ask, “how does a linear latent schedule help train it?”

  • Perceiver family.
    Perceiver / IO / AR also operate on a latent bottleneck, typically with a fixed latent size. The Super Transformer’s latent region plays a similar role, but with:
    • explicit prefix/latent/decoder regions, and
    • a latent count that can be changed over time and per sample.
  • Log-Linear Attention and ReAttention.
    Log-Linear modifies the shape of attention kernels to get long-context efficiency; ReAttention extends context at inference by retrieving top-$k$ relevant past keys/values. The Super Transformer’s contribution is orthogonal: latent state size and training dynamics. You could imagine using log-linear kernels inside the latent block or using ReAttention to pre-process a huge prefix before projecting it into latents.

We’re not trying to “beat” any of these in this blog; we’re just pointing out that once you commit to a latent workspace, scheduling its size over training is a low-overhead trick that seems to help.

Small State, Large Capacity

A nice side-effect of these experiments is a reminder that “small state” is not a death sentence if you choose the right mechanism.

  • In SSMs, a small state dimension means every piece of history has to fight for room in a single vector evolving under a fixed rule.
  • In the Super Transformer, a small number of latents gives you a handful of distinct positions that can specialize and be accessed by content via attention.

That difference matters on tasks like copy:

  • The model needs a clean way to point from “current decoder position” back to “corresponding prefix token.”
  • Latent tokens can act as a small, structured “indexing space” the model can learn to use.

Training with a larger $L$ and linearly decaying it to $L_{\min}$ is essentially saying:

“Hey model, learn your indexing trick while you’ve got a comfortable latent space, then show us you can do it in a tiny one.”

On the copy task, that works better than forcing the model to be tiny from the outset.

Limitations and Future Directions

This is a blog post and a first pass, not the final word. A few directions that would be fun to explore:

  • Beyond copy.
    Copy is a great microscope for memory, but not a full benchmark. Applying the same linear latent schedule to synthetic algorithmic tasks, long-context language, or code would be interesting.

  • Other schedules (or learned ones).
    We used a linear decay from $L_{\max}$ to $L_{\min}$. It would be easy to try cosine, piecewise, or even learned schedules that adapt based on training loss or gradient statistics.

  • Combining with other blocks.
    The three-region Super Transformer layout is compatible with SSM layers (e.g., S4/Mamba in the prefix), log-linear attention inside the latent region, and ReAttention for huge contexts. There’s a lot of room to mix and match.

  • Probing what latents actually store.
    It would be cool to freeze models at different points in the linear schedule and visualize attention maps or nearest neighbors in latent space to see how their “memory strategy” evolves as $L$ shrinks.

The main takeaway is intentionally modest:

If you already have a latent-bottleneck Transformer, trying a simple linear latent schedule is cheap and sometimes helpful, at least on memory-heavy tasks like copy.
